# 网站 API 自动化

以获得真实业务结果和可稳定复现的 API 调用链为完成标准。Browser 是发现 API 和建立鉴权的辅助工具，不是主要自动化方案；不要为了完成本可由 API 解决的任务反复点击、滚动或修补页面元素。

## 定义完成标准

任务同时满足以下条件才算完成：

1. 用户要求的查询、操作、批量处理或报表已经实际完成并验证。
2. 已用 `curl`、Python 或浏览器上下文中的直接 API 调用验证关键请求，而不是只证明页面按钮可以点击。
3. 已整理鉴权方式、接口、参数、调用顺序、分页或轮询方式和结果校验，足以据此制作一个不依赖页面点击的专用技能。

## 工作流程

1. 从用户请求中提取网站、业务目标、数据范围、筛选条件、输出格式和完成标准。次要选项采用合理默认值；关键歧义才询问。
2. 需要登录时读取并遵循 `skills/productivity/website-login/SKILL.md`。登录后返回本流程，继续获取业务接口和鉴权状态。
3. 用 Browser 打开目标页面并执行一次最短的代表性操作，例如查询一页数据、应用筛选或触发一次导出。观察它实际发送的请求，不先尝试用 Browser 重复完成全部数据或批量操作。
4. 一旦 Browser 无法可靠定位元素、无法下载、页面只展示部分数据、操作需要大量重复步骤，或一次合理尝试没有产生预期状态，立即转向 API。不要继续死磕页面交互。
5. 用 `curl` 或 Python 在页面点击之外复现请求，补齐鉴权、分页、轮询、下载和格式转换，直到得到并验证最终结果。
6. 任务完成后输出“API 复现记录”，方便后续整理为站点专用技能。

用户已经明确要求的查询、筛选、导出和普通业务操作无需逐步确认。付款、删除、发布、发消息、授权、账号变更等难以撤销且用户未明确要求的动作，才在最终提交前确认。

## Browser 已知坑点

1. **元素 ref 不稳定**：`browser_snapshot` 返回的 `e9`、`e10` 等 ref 只属于当时的页面状态。刷新、弹窗、筛选、局部渲染或 SPA 状态变化后，同一输入框的 ref 可能改变。每次页面变化后重新获取快照，并按标签、角色和附近文本重新确认元素；不要保存或复用旧 ref。ref 漂移一次或控件仍无法可靠操作时，改用 `browser_console` 定位 DOM，随后转向 API。
2. **没有网络请求拦截**：`browser_*` 不会直接给出点击产生的请求，这是发现导出和业务 API 时最大的盲区。在触发代表性操作之前，用 `browser_console` 同时 hook `window.fetch`、`XMLHttpRequest.prototype.open`、`send` 和 `setRequestHeader`，记录方法、URL、请求头和请求体；请求完成后再记录状态和响应类型。已经点击过但没有捕获请求时，先安装 hook，再只重复一次目标操作。
3. **快照信息密度低**：大表格、长列表和复杂后台页面可能只显示前几行，并出现 `... 355 more lines truncated` 一类截断。不要依赖连续滚动或重复快照拼接完整数据；立即用 `browser_console` 执行 JavaScript 提取当前 DOM、表格、页面状态或前端数据。需要全量记录时继续查找分页或列表 API。
4. **文件下载是盲区**：headless 模式下点击下载后，无法可靠确认是否下载成功以及保存位置。点击完成和页面提示都不能证明文件已经交付。应先拦截导出请求，还原创建任务、状态轮询和下载接口，取得下载 URL 后用携带鉴权的 `curl` 或 Python 保存文件，再检查路径、大小、类型和内容。
5. **纯文本模型下 `browser_vision` 会部分成功**：当模型不支持多模态输入时，`browser_vision` 每次都会报 `Model only support text input`，但截图通常已经成功拍摄并保存。该工具不会把截图路径作为独立字段返回，路径可能只出现在错误文本中。不要把这类报错理解为没有截图，也不要重复调用；从错误信息中提取实际图片路径，确认文件存在且位于当前 profile 内。如果读取图片文字即可完成任务，读取并遵循 `skills/productivity/image-text-ocr/SKILL.md`，把该路径传给 `run_ocr.py`，并告诉用户视觉分析失败后已改用本地 OCR。OCR 只能读取文字；任务仍需要物体、颜色、布局、图表趋势等视觉语义时，明确说明剩余限制。

这些限制不是继续重试 Browser 的理由。页面操作一次没有得到可靠结果，或任务涉及批量数据、分页、异步导出和文件下载时，立即使用 `browser_console` 或 `browser_cdp` 获取请求证据，并在 `curl` 或 Python 中复现。

## 观察真实请求

按可用能力组合以下方法：

- 在目标操作前用 `browser_console` 临时包装 `window.fetch` 以及 `XMLHttpRequest.prototype.open`、`send`、`setRequestHeader`，记录 URL、方法、请求头、查询参数、请求体、响应状态和响应类型；操作后恢复原函数或重新加载页面。
- 查看 `performance.getEntriesByType("resource")`、表单 action、DOM 属性、页面初始化数据、localStorage、sessionStorage、JavaScript bundle 和控制台输出。
- 当前会话支持 CDP 时，使用 Network、Runtime、DOM、Storage 等能力观察请求、响应、Cookie、iframe 和运行态。
- 识别 API 域名、认证域名、对象存储和下载域名。页面真实使用且属于同一业务流程的跨域请求和重定向也是调用链的一部分。
- 如果页面先创建任务，再轮询状态并获取下载地址，完整记录这三类请求及它们传递的任务 ID、文件 ID 或签名 URL。

优先复现页面已经成功发送的请求。可以根据前端代码、响应错误和接口返回修正参数，但不要无目的扫描无关接口或枚举其他用户、组织和资源。

## 始终携带鉴权

每次复现请求前先判断鉴权方式，不要遗漏后再盲目调接口：

- Bearer、Basic 或自定义 token 请求头。
- Cookie 会话，包括 HttpOnly Cookie。
- CSRF Cookie、请求头、隐藏字段或页面初始化 token。
- `Origin`、`Referer`、tenant、workspace、organization 等上下文标识。
- 签名参数、时间戳、nonce 和短期下载 URL。

优先在浏览器上下文中直接 `fetch`，确认请求在当前会话内能够成功。随后使用 `curl` 或 Python 复现；需要时从 `browser_console`、存储或 CDP 取得当前任务所需的 Cookie、token 和 CSRF 信息。`401`、`403`、CSRF 错误或签名过期通常意味着鉴权或上下文不完整，应先与浏览器成功请求逐项对照。

登录密码不要交给终端。Cookie、token 和完整认证头可以在当前任务内使用，但不得出现在最终回复和普通日志中；通过环境变量、内存、标准输入、Cookie jar，或 `/opt/data/tmp` 中权限为 `0600` 的临时文件传递，并在任务结束后删除。整理复现方法时使用 `$TOKEN`、`$COOKIE_JAR`、`$CSRF_TOKEN` 等占位符，同时说明如何从登录会话刷新这些值。

## 使用 curl 和 Python

- 单个或少量请求优先用 `curl`，保留 HTTP 状态和错误正文，正常验证 TLS，并显式带上请求头、Cookie、查询参数和请求体。
- 多页数据、会话复用、异步轮询、流式下载、签名计算和格式转换使用 `/opt/hermes/.venv/bin/python`。优先使用标准库和现有环境，需要时使用现有 `requests` 等库。
- 用 Cookie jar 或 `requests.Session` 保持会话。后续请求依赖前一步返回的 token、任务 ID、游标或下载 URL 时，在代码中显式传递。
- 分页必须确认页码、游标、总数和终止条件；异步任务按状态与建议间隔轮询，并设置合理截止时间；`429` 按 `Retry-After` 等待，临时网络错误和 `5xx` 使用有上限的退避重试。
- 数据量较大时直接流式写入文件，不把完整响应放进对话上下文。先保存原始响应或原始文件，再进行可重复的清洗和转换。

关键 API 必须至少成功复现一次。只有 Browser 中能成功、离开页面就失败的请求还不算稳定；继续补齐鉴权、上下文和调用顺序，或准确记录无法脱离浏览器获取的短期凭据及刷新步骤。

## 导出和交付

1. 先确认筛选范围，再判断导出流程是直接文件、异步任务、分页 API 或签名下载地址。
2. 优先保存网站原始文件；只能取得结构化数据时，用 Python 重建用户需要的 CSV、JSON 或其他合理格式。
3. 最终文件保存到用户指定的 `/opt/data` 内路径；未指定时保存到 `/opt/data/.dock/shared`。临时脚本、Cookie jar、会话材料、分片下载和缓存写入 `/opt/data/tmp`，完成后清理。
4. 验证文件存在且非空、格式或文件签名正确、分页完整、记录数和筛选范围合理，并尽可能与页面汇总或抽样记录交叉核对。
5. 用户通过消息通道要求发送、交付或下载文件时，确认文件真实存在后，在最终回复中单独输出一行 `MEDIA:/opt/data/文件路径`。

## 输出 API 复现记录

任务成功时，在最终回复中整理以下内容；内容较长时同时写入 `/opt/data/.dock/shared` 下的 Markdown 文件：

1. **鉴权**：使用 token、Cookie、CSRF 或签名中的哪些机制，值从哪里取得、何时过期、如何刷新；只写变量名和脱敏占位符。
2. **调用顺序**：按 `1 → 2 → 3` 列出请求，说明前一步哪些返回值传给下一步。
3. **接口清单**：每个接口的域名、方法、路径、必要请求头、Cookie、查询参数、请求体、成功状态和关键响应字段。
4. **分页与异步流程**：游标、页码、任务状态、轮询间隔、终止条件和下载步骤。
5. **复现示例**：提供脱敏且可运行的 `curl` 或 Python 示例，使用环境变量和占位符，不嵌入当前会话秘密。
6. **结果校验**：记录数、文件类型、关键汇总、抽样验证和已知限制。

这份记录应描述真实验证过的请求，不把猜测写成结论。除非用户明确要求，不自动创建或安装新的站点专用技能，但要让记录足以作为后续技能的输入。

## 及时停止

Browser 失败本身不是停止条件。只有必须由用户完成验证码、硬件密钥或客户端证书，当前账号被明确拒绝且重新获取鉴权后仍无权限，目标存在关键歧义，或 Browser、前端代码和 API 路径都已检查且没有新的可验证方案时才停止。

停止时说明已还原的接口、鉴权进展、准确错误、尚缺的条件和可继续的方向，不要只说“网站自动化无法完成”。
